## Data Dictionary

### Log Fields (dev-logs/)

Every JSON log record contains:

```json
{
  "timestamp": "ISO 8601 UTC",
  "service": "gateway|fraud-scoring|validation-enrichment|aml-compliance|routing-execution|settlement|audit",
  "correlationId": "Payment tracing ID — use to join events across services",
  "paymentId": "ISO 20022 unique payment identifier",
  "event": "payment.initiated|fraud.evaluated|validation.complete|aml.hit|routing.selected|settlement.settled|audit.recorded",
  "level": "INFO|WARN|ERROR",
  "message": "Human-readable event description",
  "paymentAmount": "decimal with 2 places",
  "decision": "PASS|BLOCK|HIT|SETTLE",
  "latencyMs": "Processing time in milliseconds",
  "stage": "0|1|2|3|4|5|6|7 (payment pipeline stage)"
}
```

### Batch Test Output (batch_100k_eval_run.log)

```
Batch    — Batch number (1–200)
Sent     — Payments sent in this batch (500 per batch)
OK       — Payments accepted (HTTP 202)
Err      — Connection errors
RL       — Rate-limited (HTTP 429)
req/s    — Requests per second
Done     — Cumulative total through this batch
```

### Code Graph (graphify-out/)

1,162 Java classes, methods, interfaces from the codebase, with edges for:
- Calls (A → B means A calls B)
- Implements (A → I means A implements interface I)
- Produces-to (A → Topic means A emits Kafka topic)
- Consumes-from (A → Topic means A subscribes to Kafka topic)

Use this to validate LLM claims in root cause analysis (e.g., "Does `KafkaEventPublisher` really have a `publishFallback()` method?")

## How to Regenerate Any Artifact

All artifacts are generated by the ClearFlow project itself. To recreate:

```bash
# Start all services
bash clearflow-start.sh

# Run the 100K test
python3 batch_100k.py > batch_100k_eval_run.log 2>&1

# Logs will appear in dev-logs/ as tests run
# Elasticsearch indices available at :9200/clearflow-*

# Code graph is already in the repo
cd graphify-out/
# See GRAPH_REPORT.md for community structure

# Grafana dashboards are provisioned automatically
# Available at http://localhost:3001
```

## Validation Checklist for Reviewers

- [ ] All `paymentId` values are valid UUIDs
- [ ] All `correlationId` values match across services for the same payment
- [ ] Log timestamps are monotonically increasing within a service
- [ ] Acceptance rate reported in paper matches `OK / Sent` calculation from batch_100k_eval_run.log
- [ ] p99 latency is >= p95 latency (sanity check on percentile math)
- [ ] Every settled payment has a debit and credit entry (double-entry property)
- [ ] Hash chain is continuous (no gaps, no unseen paymentIds)
