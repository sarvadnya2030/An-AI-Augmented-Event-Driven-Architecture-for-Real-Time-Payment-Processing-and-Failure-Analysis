🔍 ROOT CAUSE ANALYSIS

  Visible In:
    ├─ ELK: 92% of logs from gateway (obvious bottleneck)
    ├─ MCP: Batch 1 = 32%, Batch 2 = 0% (cascade point)
    ├─ Graphify: Circuit breaker failure boundary
    └─ Dashboard: All three visualized together

  Root Cause:
    ├─ Primary: ActiveMQ connection pool exhaustion (50 limit)
    ├─ Trigger: Batch 1 requires 160+ concurrent connections
    ├─ Cascade: Circuit breaker opens → all downstream blocked
    └─ Result: 0% acceptance for batches 2-10

  Why 2GB Heap + G1GC Wasn't Enough:
    ✅ Prevented OutOfMemory (no crashes)
    ✅ Maintained low latency (194ms max)
    ✓ Application stayed running
    
    ❌ Didn't expand connection pool (still 50)
    ❌ Didn't improve broker throughput
    ❌ Didn't tune circuit breaker (binary open/close)
    └─ All require infrastructure changes, not JVM tuning

  Recovery Actions (Priority):
    1. Increase ActiveMQ pool (50 → 100+)
    2. Increase broker throughput
    3. Tune circuit breaker (graceful degradation)
    4. Add monitoring & alerting
